## flux.toml

The single config file for a Flux project. Everything a project needs to declare lives here.

```toml
[project]
name    = "my-app"
id      = "uuid-assigned-on-first-deploy"   # filled in by flux deploy

[runtime]
timeout_ms      = 30_000    # default per function
memory_mb       = 128
hot_reload      = true      # watch functions/ for changes in dev mode

[db]
migrations_dir  = "schemas"  # path to user SQL migrations

[[cron]]
function = "cleanup_sessions"
schedule = "0 * * * *"       # every hour

[[cron]]
function = "send_digest"
schedule = "0 9 * * 1"       # Mondays at 9am UTC

[middleware]
global = ["auth"]            # applied to every function

[[middleware.groups]]
name       = "admin"
middleware = ["auth", "require_admin"]
functions  = ["delete_user", "ban_user"]

[rate_limit]
requests_per_minute = 1000   # project-wide default
```

## .env File

The `.env` file at the project root is loaded by `flux dev` and `flux run`. Variables here take
precedence over the embedded Postgres defaults but are overridden by actual shell environment
variables.

```bash
# Connect to an external Postgres instead of the embedded one.
# If set, flux dev skips starting embedded Postgres entirely.
DATABASE_URL=postgres://user:pass@localhost:5432/myapp

# Any env var your functions need at runtime.
OPENAI_API_KEY=sk-...
STRIPE_SECRET_KEY=sk_live_...
```

**Commit `.env.example`, never `.env`.**

## .flux/ Directory

This entire directory is local to the developer's machine. It should be in `.gitignore`:

```gitignore
.flux/
```

It is recreated automatically by `flux dev` and `flux generate`. The only file you might
occasionally want to inspect (not edit) is `.flux/manifest.json` for debugging.

## Recommended .gitignore

```gitignore
# Flux local state and generated files
.flux/

# Local environment overrides вЂ” commit .env.example, not .env
.env
```

## Context Resolution

When any CLI command needs to talk to a Flux server (e.g. `flux generate`, `flux invoke`,
`flux db push`, `flux trace`, `flux deploy`), the server endpoint and API key are resolved in
this order:

| Priority | Source |
|----------|--------|
| 1 | `--context <name>` CLI flag |
| 2 | `FLUX_CONTEXT` environment variable |
| 3 | `.flux/contexts.toml` in project root (local project override) |
| 4 | `~/.flux/contexts.toml` active context (global) |
| 5 | `http://localhost:4000` (default, no auth required) |

Manage contexts with:

```bash
flux link prod https://myapp.com --key sk_live_xxx   # add/update
flux use prod                                         # switch active context
flux context                                          # show current context
flux unlink staging                                   # remove a context
```

## DATABASE_URL Priority (flux dev)

When `flux dev` starts, it resolves the database connection in this order:

| Priority | Source |
|----------|--------|
| 1 | `DATABASE_URL` shell environment variable |
| 2 | `DATABASE_URL` in `.env` file at project root |
| 3 | Embedded Postgres (auto-downloaded, zero-config) |

Levels 1 and 2 skip embedded Postgres entirely вЂ” useful when you want to develop against
Neon, Supabase, or a local Postgres you manage yourself.

## Config Priority Chain (runtime)

All configuration follows this precedence (highest to lowest):

| Priority | Source |
|----------|--------|
| 1 | CLI flag (e.g. `--key`, `--endpoint`) |
| 2 | Shell environment variable |
| 3 | `.env` file in project root |
| 4 | `flux.toml` in project root |
| 5 | Flux server DB (`flux.project_config` table) |
| 6 | Built-in defaults |
