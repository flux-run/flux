# Folder Structure

This document describes both the repository layout and the project layout generated by `flux init`.

## Repository Layout

The repo is split into product-facing subsystems:

```text
flowbase/
├── api/              # operator-facing APIs
├── cli/              # developer and operator CLI
├── dashboard/        # dashboard UI
├── data-engine/      # database execution and mutation recording
├── docs/             # product and architecture docs
├── gateway/          # ingress, routing, middleware, auth
├── queue/            # async jobs and retries
├── runtime/          # user code execution
├── scaffolds/        # project and function templates
├── schemas/          # framework-level SQL and schema assets
├── server/           # single-binary deployment target
└── shared/           # shared contracts and libraries
```

The repo preserves separate component crates because the boundaries matter architecturally, even though the deployment story is a monolith.

## Project Layout

A Flux project looks like this:

```text
my-app/
├── flux.toml
├── .env.example
├── .env.local                # gitignored
├── functions/
│   └── create_user/
│       ├── index.ts
│       └── flux.json
├── schemas/
│   └── users.schema.ts
├── middleware/
│   └── auth.ts
├── queues/
│   └── email.queue.toml
└── .flux/                    # generated files and local state
```

## Meaning Of Each Project Directory

- `flux.toml` - project-level configuration and runtime intent
- `functions/` - user-defined function handlers
- `schemas/` - database schema or migration definitions
- `middleware/` - request or route middleware
- `queues/` - queue definitions and worker settings
- `.env.example` - committed environment template
- `.env.local` - local secrets and overrides
- `.flux/` - generated files, manifests, dev state, and caches

## `flux.toml`

`flux.toml` is the project's main declarative config file.

`flux.toml` covers:

- project identity
- local development ports and behavior
- runtime limits
- middleware defaults
- queue and schedule configuration
- observability defaults

The user-facing principle stays the same:

- one project config file
- obvious defaults
- minimal hidden state

## `.flux/`

`.flux/` is local runtime state and generated artifacts. It is normally gitignored.

Typical contents:

- generated manifest and type files
- local development metadata
- embedded Postgres state in dev mode
- local CLI configuration

The goal is that users rarely edit `.flux/` manually.

## Environment Files

Recommended pattern:

- commit `.env.example`
- keep `.env.local` and other local secret files out of version control
- allow `DATABASE_URL` to override embedded development database behavior

## Design Rule

The folder structure makes the product legible:

- a developer immediately sees that Flux covers functions, data, async work, and debugging
- a contributor immediately sees how the repo maps to the product architecture
