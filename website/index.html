The Debugging Workflow

From alert to root cause in 30 seconds.

A user reports "signup failed". You have a request ID from flux tail. One more command and you know exactly what happened.

Step 1 — spot the failure

flux tail 
$ flux tail

  Streaming live requests…

  ✔  POST /signup      201   88ms  req:4f9a3b2c
  ✔  GET  /users       200   12ms  req:a3c91ef0
  ✗  POST /signup      500   44ms  req:550e8400
     └─ Error: Stripe API timeout

Step 2 — understand it

flux why 550e8400 
$ flux why 550e8400

  ROOT CAUSE
  Stripe API timeout after 10s

  LOCATION
  payments/create.ts:42

  DATA CHANGES
  users id=42  plan: free → null  (rolled back)

  SUGGESTION
  → Add 5s timeout + idempotency key retry

The Difference

You shouldn't need 5 tools to debug one request.

Traditional backends scatter evidence across logs, metrics, and traces — each in a different tool without shared context. Fluxbase records everything in one place, queryable by request ID.

Traditional backend debugging
✗ logs — scattered across services
✗ metrics dashboard — no request context
✗ trace UI — requires manual SDK instrumentation
✗ DB console — query by query
✗ queue monitor — separate tool
Fluxbase
✓ flux why <id> — root cause, one command
✓ flux trace <id> — full span tree, latencies
✓ flux state history — every row mutation
✓ flux incident replay — safe re-execution
✓ flux bug bisect — which commit broke it
Capabilities

Production debugging, not just monitoring.

Monitoring tells you something is wrong. Fluxbase tells you why, shows you what changed, and lets you replay it.

🔍

Time-Travel Debugging flux trace debug

Step through a production request span by span. See the exact input, output, and state at every point in execution.

📜

Mutation History flux state history

Every database write is logged with its request ID. See the full history of any row — what changed, when, and which request caused it.

♻️

Incident Replay flux incident replay

Re-run a production time window with side-effects disabled. Test your fix against exactly the requests that caused the incident.

🔎

Regression Detection flux bug bisect

Binary-searches your git history to find the first commit where a request started failing. Like git bisect, but for production behaviour.

🛡️

Deterministic Execution recorded by default

Every request captures a complete trace automatically — no instrumentation, no SDKs, no config. The runtime produces the trace.

🔷

Observable by Construction zero config

Gateway, functions, database queries, tool calls, async jobs — every layer emits spans automatically. flux trace reconstructs the full picture.

How It Works

One request ID covers the entire stack.

Client → Gateway → Runtime → Data Engine → Your PostgreSQL. Every hop records a span. flux trace reassembles them in order.
