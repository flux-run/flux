How It Works

One request ID.
The entire stack.

Fluxbase adds a recording and replay layer around your execution stack. Every layer — from the gateway to the database — emits structured spans tied to one request ID. The CLI reassembles them on demand.

Try it → See the features
The Architecture

Five layers. One trace.

Every layer is instrumented at the runtime level — no application-level tracing hooks needed. The span data is stored in the Data Engine alongside mutation logs.

Step by Step

What happens when a request runs.

1

Request capture

When a request arrives at the Gateway, Fluxbase assigns it a globally unique request ID (UUID v4). This ID is propagated via internal headers to every downstream service. The gateway records auth result, rate-limit decision, matched route, and timing as the first span.

gateway → span 
# Gateway emits:
{
  request_id: "4f9a3b2c",
  span: "gateway",
  method: "POST",
  path: "/create_user",
  auth: "ok",
  duration_ms: 3
}
2

Function execution

The Runtime receives the request with the forwarded request ID. It executes your TypeScript function in a sandboxed V8 isolate. Every ctx.db, ctx.tool, and ctx.workflow call is intercepted and recorded as a child span under the function span.

runtime → spans 
# Runtime emits per call:
{
  request_id: "4f9a3b2c",
  span: "create_user",
  children: [
    { span: "db:select(users)", 11ms },
    { span: "db:insert(users)", 14ms }
  ],
  duration_ms: 81
}
3

Mutation logging

Every database write goes through the Data Engine, which applies schema validation, column policies, and row-level security before executing the SQL. After execution, it writes a mutation record: which table, which row, old value, new value, and the request ID that caused it.

Invariant: All database writes must pass through the Data Engine for mutation history to remain complete. Writes made directly to Postgres — bypassing the Data Engine — are not recorded.

data engine → mutation log 
# Mutation record:
{
  request_id: "4f9a3b2c",
  table: "users",
  row_id: 42,
  operation: "insert",
  data: { email: "a@b.com", plan: "free" },
  timestamp: "2026-03-10T14:22:01Z"
}
4

Trace graph

All spans for a request ID are stored in an ordered graph. flux trace <id> retrieves them and renders the full tree — gateway, function, database queries, tool calls — in execution order with latencies.

trace store → rendered 
$ flux trace 4f9a3b2c

  gateway           3ms
  create_user      81ms
    db:select       11ms
    db:insert       14ms
  send_welcome  async → queued

  total: 98ms
5

Deterministic replay

Because every span includes its full input and output, any request can be replayed against the current code. flux incident replay re-executes with side-effects disabled. flux bug bisect replays across your git history to find regressions.

What replay guarantees: database state transitions are reproduced exactly. External side effects — emails, webhooks, third-party API calls — are skipped during replay. Non-deterministic values like random() or Date.now() inside your own code may differ; the recorded data state will not.

replay — side-effects off 
$ flux incident replay 14:00..14:05

  hooks: off · events: off · cron: off
  db writes: on · mutation log: on

  ✔ 22/23 passing
  ✗  req:550e8400 still fails
     Stripe timeout at payments/create.ts:42
Performance

Typical overhead: 1–3 ms per request.

Span recording is in-memory and non-blocking. Mutation logging shares the same Postgres transaction as your own write. No synchronous tracing pipeline. No global locks.

per-request overhead breakdown 
# Time added by Fluxbase per request

  span recording      ~0.3 ms   in-memory, async write
  mutation logging    ~0.7 ms   one row, same tx as user write
  trace storage       ~0.4 ms   fire-and-forget, non-blocking
                      ─────────
  total               ~1–3 ms   p95 overhead

# Your user write and the mutation diff
# commit in the same Postgres transaction.

  user write     ─────────────────► Postgres
  mutation diff  ─── same tx  ────► trace store

# Slow trace write does not delay HTTP response.
⚡ In-memory spans

Spans are assembled in-memory during execution. A single batch insert commits them after the response is sent.

📝 Same-transaction diffs

Mutation diffs are appended in the same Postgres transaction as the user write — no extra round-trip to the database.

🔥 Fire-and-forget trace write

The span batch insert is non-blocking. A slow trace write cannot delay the HTTP response to your client.

🚫 No cross-request locking

Each request writes its own rows independently. There is no global lock, no shared state between concurrent requests.

Technology

Open foundations, high-performance core.

Every service is written in Rust for predictable latency and memory safety. Your TypeScript functions run in Deno V8 isolates. Your data stays in standard Postgres.

🛡️
Gateway
Rust (Axum)

Auth, rate limit (per-tenant token bucket), query guard, semaphore budget. Routes requests to Runtime or Data Engine.

⚡
Runtime
Rust + Deno V8

Executes TypeScript in sandboxed V8 isolates per tenant. Warm isolates for low latency; per-tenant affinity to prevent cross-tenant heap contamination.

🗄️
Data Engine
Rust (Axum)

Query compiler (JSON → SQL), column policies, row-level security, BYODB (Bring Your Own Database). Mutation log writer and explain endpoint.

📬
Queue
Rust

Durable async job queue. Stores job payloads in Postgres. Workers poll and execute functions from the Runtime. Fully traced.

🔌
API
Rust (Axum)

Management API: deploy functions, manage schemas, API keys, tenant config. Consumed by the CLI and Dashboard.
