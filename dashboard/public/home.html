The Difference

Traditional backends discard execution. Fluxbase keeps it.

Every backend request normally runs and disappears — logs printed, memory freed, context gone. Fluxbase makes execution permanent. Every request becomes a queryable record.

Traditional backend: execution disappears
✗ request runs → logs printed → context gone
✗ incident happens → reconstruct from fragments
✗ debug production → reproduce locally (or not)
✗ data changes → no record of what caused it
✗ regression appears → grep through git history
Fluxbase: execution is inspectable history
✓ flux tail — stream live execution history
✓ flux why <id> — root cause, one command
✓ flux incident replay — re-run production safely
✓ flux state history — every mutation, every request
✓ flux bug bisect — find exact breaking commit
The Shift

A new execution model.

Backend systems have always been ephemeral by default — executions run and vanish. Fluxbase inverts this. Execution becomes the permanent artifact, not an afterthought.

Traditional backend
📨 Request arrives
⚙️ Code runs
🗒️ Logs printed
💨 Execution disappears
Fluxbase
📨 Request arrives
⚙️ Code runs
🗃️ Execution recorded
♻️ Replayable
⚖️ Diffable
🔍 Blameable
🔬 Step-through debuggable

This is why flux why can answer in one command what used to take hours of log archaeology.

Capabilities

Production debugging, not just monitoring.

Monitoring tells you something is wrong. Fluxbase tells you why, shows you what changed, and lets you replay it.

🔍

Time-Travel Debugging flux trace debug

Step through a production request span by span. See the exact input, output, and state at every point in execution.

📜

Mutation History flux state history

Every database write is logged with its request ID. See the full history of any row — what changed, when, and which request caused it.

♻️

Incident Replay flux incident replay

Re-run a production time window with side-effects disabled. Test your fix against exactly the requests that caused the incident.

🔎

Regression Detection flux bug bisect

Binary-searches your git history to find the first commit where a request started failing. Like git bisect, but for production behaviour.

🛡️

Deterministic Execution recorded by default

Every request captures a complete trace automatically — no instrumentation, no SDKs, no config. The runtime produces the trace.

🔷

Observable by Construction zero config

Gateway, functions, database queries, tool calls, async jobs — every layer emits spans automatically. flux trace reconstructs the full picture.

Why It Works

Observability is not a feature. It is how the runtime works.

There is no "add tracing later" checkbox. Every execution is recorded at the infrastructure level — not by SDKs, not by your code.

📝

Every request is recorded.

The gateway captures inputs, outputs, and metadata for every HTTP request. No SDK. No instrumentation. No config.

📃

Every mutation is logged.

When your function writes to PostgreSQL, the Data Engine intercepts it and stores the row diff with its request_id. Your database is auditable by default.

🔬

Every execution span is traced.

Gateway, runtime, DB queries, tool calls, async jobs — each layer emits spans automatically. flux trace reassembles the full picture from a single ID.

♻️

Production can be replayed safely.

Because inputs and state are captured, any time window can be re-executed against your current code. Side-effects are disabled. Your fix is tested against real production traffic.

How It Works
