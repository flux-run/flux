Deployment

Fluxbase can run as a managed cloud service or as a fully self-hosted stack on your own infrastructure. Both options use the same CLI and the same BYODB (Bring Your Own Database) model.

Deployment options

Cloud (managed)

Fluxbase runs the infrastructure at fluxbase.co. You install the CLI, flux login, and deploy. Zero infrastructure to manage.

  • Deploy in ~20 seconds
  • Automatic scaling
  • Your Postgres, connected via DATABASE_URL
  • Trace data stored in your own DB
Self-hosted

All Fluxbase services are open source. Run the full stack with Docker Compose on your own machines, VMs, or Kubernetes cluster.

  • Full control over all services
  • Your network, your region, your certs
  • Same BYODB model
  • Same CLI, pointed at your endpoint

Cloud setup

The managed cloud option requires no infrastructure. Install the CLI, authenticate, connect your Postgres, and deploy.

cloud: deploy in 20 seconds 
# Install CLI
$ curl -fsSL https://fluxbase.co/install | bash

# Authenticate
$ flux login
  ✔  Logged in as alice@example.com

# Deploy your functions
$ flux deploy
  ✔  create_user  → gw.fluxbase.co/create_user
  ✔  list_users   → gw.fluxbase.co/list_users
  ✔  Deployed in 18s

# Start debugging
$ flux tail

Trace and mutation data is written to a fluxbase_trace schema inside the Postgres database you provide. It is never stored on Fluxbase infrastructure.

Self-hosted setup

The self-hosted stack runs five Rust services plus Postgres. All source code is in the fluxbase/fluxbase repository.

self-hosted: docker compose up 
# Clone the repo
$ git clone https://github.com/fluxbase/fluxbase
$ cd fluxbase

# Configure environment
$ cp .env.dev.example .env.dev
$ $EDITOR .env.dev   # set DATABASE_URL, etc.

# Start all services
$ flux stack up --build

  ✔  db            postgres:16  :5432
  ✔  api           :8080
  ✔  gateway       :8081
  ✔  data-engine   :8082
  ✔  runtime       :8083
  ✔  queue         :8084

# Point CLI at local stack
$ flux login --endpoint http://localhost:8080

The CLI is identical in both modes. Point it at your local gateway with --endpoint or the FLUX_ENDPOINT environment variable.

Services

Port Service Tech Role
:8080 API Rust (Axum) Management plane — deploy functions, manage API keys, tenant config, schema definitions. Consumed by the CLI and Dashboard.
:8081 Gateway Rust (Axum) Public execution edge — auth, rate limiting, request routing. Assigns request IDs and emits the first span. Horizontally scalable.
:8082 Data Engine Rust (Axum) Query compiler and mutation logger — validates writes, applies row-level security and column policies, writes mutation diffs in the same transaction as user writes.
:8083 Runtime Rust + Deno V8 TypeScript execution — runs your functions in sandboxed V8 isolates per tenant. Warm isolates for low latency. All ctx.db / ctx.tool calls are intercepted here.
:8084 Queue Rust Async job queue — stores job payloads in Postgres, polls and executes async functions via the Runtime. Fully traced.

All services communicate over internal HTTP. The only public-facing port is the Gateway (:8081).

BYODB — Bring Your Own Database

Both cloud and self-hosted use the same database model: you provide the Postgres instance. Fluxbase never stores application data on its own infrastructure.

BYODB — your postgres, always 
# Point Fluxbase at your existing Postgres
# in .env.dev or cloud project settings:

  DATABASE_URL=postgresql://user:pass@your-db.host:5432/myapp

# Fluxbase creates two schemas:

  public          ← your application tables (unchanged)
  fluxbase_trace  ← spans, mutations, request log

# Your tables are never modified.
# Trace data lives in its own schema.
# You control backups, region, encryption.

Fluxbase creates one additional schema (fluxbase_trace) in your Postgres for spans, mutation diffs, and the request log. Your application tables are never modified. The trace schema can be in the same cluster or a separate one.

What this means for compliance and residency

  • Data residency is determined by where you host Postgres — your choice of cloud and region.
  • Encryption at rest follows your Postgres configuration.
  • Backups are your responsibility (same as any other data in your DB).
  • Deletion — dropping the fluxbase_trace schema removes all trace data immediately.

Common questions

Can I run only some services?

Yes, with some caveats. The Runtime, Data Engine, and Gateway are co-dependent — removing any one breaks the execution and mutation recording pipeline. The Queue service is optional if you don't use async functions. The API service is optional for read-only CLI operations but required for deploys and key management.

Does self-hosted support multi-tenancy?

Yes. Tenant isolation is enforced inside the services via tenant_id on every table and V8 isolate affinity in the Runtime. The cloud and self-hosted stacks use identical isolation logic.

What about auth in self-hosted mode?

The managed cloud uses Firebase Auth. In self-hosted mode you can configure an alternative OIDC provider, or use API key auth with FLUX_API_KEY for CLI access. See the .env.dev.example in the repository for all configuration options.

Is the self-hosted stack production-ready?

The self-hosted Dockerfiles are the same images used in the managed cloud. For production we recommend running the Gateway and Runtime in at least two replicas, with Postgres behind a connection pooler (e.g. PgBouncer).

In short

Cloud or self-hosted, the guarantees are identical: your Postgres, your data, your infrastructure. Fluxbase records the execution history in a schema you own and can delete at any time.
